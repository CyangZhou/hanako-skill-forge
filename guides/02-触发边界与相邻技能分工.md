# 触发边界与相邻技能分工

## 本技能负责什么

`hanako-skill-forge` 负责把技能做成长期可维护的工程对象，重点处理：

- 技能新建、封装、重构；
- 技能入口、路由、模块、模板的结构治理；
- 主来源唯一化；
- 维护、扩展、自愈链补齐；
- 验收口径、证据口径、交付口径统一；
- 旧残留、断链、漂移、失序收口。

## 本技能不负责什么

- 普通代码 bug 排查；
- 单句文案润色；
- 单文件小改；
- 泛化的 AI 工作流头脑风暴；
- 与技能结构无关的日常执行任务。

## 与相邻技能的边界

### 与 `skill-creator` 的边界

优先用 `skill-creator` 的场景：
- 要做技能测试集、评估集、benchmark；
- 要对 skill description 做触发优化；
- 要做新旧版本对比评测；
- 要跑多轮 eval 并根据结果迭代；
- 要把触发样例真正跑成评测结果，而不是只维护样例文档。

优先用 `hanako-skill-forge` 的场景：
- 要把技能结构做薄、做清楚、做长期可改；
- 要统一入口、模块、模板、验收链；
- 要补维护 / 扩展 / 自愈分支；
- 要清理旧残留、断链、职责重叠。

协作方式：
- 先用 `hanako-skill-forge` 收口结构；
- 再用 `skill-creator` 做评测与触发优化；
- 若评测暴露结构问题，再回到本技能修结构。

### 与 `ai-workflow-forge` 的边界

优先用 `ai-workflow-forge` 的场景：
- 还在聊一个模糊想法；
- 还没确定是否值得做成技能；
- 重点是工作流收敛，而不是技能工程落地。

优先用 `hanako-skill-forge` 的场景：
- 已决定做成技能；
- 已进入技能目录、路由、模块、模板、验收施工；
- 已经出现结构问题，需要正式收口。

协作方式：
- `ai-workflow-forge` 负责把想法变成方案；
- 本技能负责把方案变成可维护技能。

### 与普通机械施工的边界

以下情况不要触发本技能：
- 只是改一个文件里的几句话；
- 只是修一个明确的小 bug；
- 只是执行一次命令；
- 不需要调整技能结构、主来源或验收口径。

## 触发判断口诀

- 涉及技能工程收口，用本技能；
- 涉及技能评测优化，用 `skill-creator`；
- 涉及模糊工作流构思，用 `ai-workflow-forge`；
- 只是小修小补，不启用元技能。
