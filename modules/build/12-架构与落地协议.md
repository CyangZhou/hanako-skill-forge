# 12-架构与落地协议.md

## 标准落地顺序

1. 先定原则与需求卡；
2. 再定路由；
3. 再定模块；
4. 再定模板；
5. 最后才判断脚本是否有必要；
6. 完成后进入验收与发布。

## 标准骨架

```text
skill-name/
  SKILL.md
  guides/
  modules/
    route/
    build/
    maintenance/
    extension/
    recovery/
    review/
    templates/
  scripts/
  schemas/
```

## 脚本门禁

只有满足以下任一条件，才允许新增脚本：

- 需要自动巡检文档引用；
- 需要自动扫旧术语或坏味道；
- 需要自动生成结构化输出；
- 需要把维护或自愈动作收成可重复命令。

如果只是解释规则、列清单、写模板，不要上脚本。

## 落地规则

- `SKILL.md` 只负责入口、能力、路由、纪律；
- `guides/*.md` 负责原则和设计口径；
- `modules/*/*.md` 负责具体规则和分支；
- `scripts/*.py` 只做巡检、校验、收口，不做隐形主工作流；
- `schemas/*.json` 只放结构化动作或契约。

## 表达规则

- 只保留功能必要语句；
- 删除花哨开场、口号式表达、重复强调；
- 规则要能直接指导动作，不能只营造气氛；
- 维护、扩展、自愈要写成“以后改这个技能时怎么走”的分支，不要写成技能运行时默认执行的内部功能；
- 能用短句说清的，不写成长段。

## 最小交付

- 可触发的 `SKILL.md`；
- 需求卡与总控路由；
- 架构与落地规则；
- 维护、扩展、自愈、验收分支；
- 至少一份维护脚本入口；
- 交付摘要。
